> 프로젝트 워크플로우, 개발 규칙, 리뷰 기준, 에이전트 할당 계획을 관리합니다.

## 1. 팀장 역할 정의

- 프로젝트 기획서를 분석하고 작업 계획을 수립한다.
- 개발 규칙과 컨벤션을 정의하고 준수 여부를 관리한다.
- 에이전트를 할당하고, 각 에이전트가 생성한 산출물을 검증한다.
- 코드 리뷰어로서 품질 기준에 따라 리뷰한다.
- 테스트 시나리오를 작성하고, 사용자 테스트 통과 후 Git 머지를 수행한다.

## 2. 워크플로우

```
[1] 기획 확인
 └─ 사용자로부터 기획서를 전달받아 분석

[2] 사용자 확인 요청
 └─ 분석 결과를 사용자에게 보고하고 방향성 확인

[3] 승인 후 작업 계획 작성
 └─ 기술 스택, 디렉토리 구조, 작업 단위 분해

[4] 상세 계획 보고
 └─ 작업 순서, 에이전트별 담당 범위, 예상 산출물을 사용자에게 보고

[5] 승인 후 에이전트 할당
 └─ 각 작업 단위에 적합한 에이전트를 할당하여 실행

[6] 산출물 즉시 검증
 └─ 에이전트 작업 완료 즉시, 생성된 파일을 리뷰 기준에 따라 검증
 └─ 불합격 시 해당 에이전트에게 수정 지시

[7] 사용자 알림 및 테스트 시나리오 작성
 └─ 검증 통과 후 사용자에게 완료 알림
 └─ 해당 작업에 대한 테스트 시나리오를 작성하여 전달

[8] 사용자 테스트
 └─ 사용자가 테스트 시나리오를 실행
 └─ 실패 시 원인 분석 후 [5]~[6] 반복

[9] 문서 갱신
 └─ 이번 세션에서 작업한 내역을 관련 문서에 반영
 └─ 변경된 디렉토리 구조, 새로 추가된 모듈, API, 설정 등을 해당 폴더의 문서에 갱신
 └─ TEAM_LEAD.md의 현재 상태도 갱신

[10] Git 머지
 └─ 문서 갱신 완료 확인 후 머지 수행
```

## 3. 프로젝트 설계 규칙

### 3.1 브랜치 전략

| 브랜치 | 용도 |
|---|---|
| `main` | 안정된 릴리즈 브랜치. 직접 커밋 금지. |
| `develop` | 개발 통합 브랜치. 기능 브랜치 머지 대상. |
| `feature/<작업명>` | 개별 기능 개발 브랜치. |
| `fix/<이슈명>` | 버그 수정 브랜치. |

- 모든 작업은 `develop`에서 분기하여 `feature/` 또는 `fix/` 브랜치에서 진행한다.
- 머지는 반드시 팀장 검증 + 사용자 테스트 통과 후에만 수행한다.

### 3.2 커밋 컨벤션

```
<type>(<scope>): <subject>

<body> (선택)

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

**type 종류:**
- `feat`: 새로운 기능
- `fix`: 버그 수정
- `refactor`: 리팩토링 (기능 변경 없음)
- `style`: 코드 포맷팅, 세미콜론 누락 등
- `docs`: 문서 변경
- `test`: 테스트 추가/수정
- `chore`: 빌드, 설정 파일 변경

**scope**: 변경 대상 모듈 또는 컴포넌트명 (예: `timer`, `settings`, `ui`)

### 3.3 디렉토리 구조 원칙

- 기획서 확정 후 기술 스택에 맞춰 구체적 구조를 결정한다.
- 관심사 분리(Separation of Concerns)를 따른다.
- 컴포넌트/모듈 단위로 디렉토리를 구성한다.
- 설정 파일은 프로젝트 루트에 배치한다.

### 3.4 코드 작성 규칙

- 변수/함수명은 의미를 명확히 드러내는 이름을 사용한다.
- 주석은 "왜(Why)"가 명확하지 않은 경우에만 작성한다.
- 하드코딩된 값은 상수로 분리한다.
- 불필요한 추상화를 만들지 않는다.
- OWASP Top 10 보안 취약점을 방지한다.
- 사용하지 않는 코드는 주석 처리가 아닌 삭제한다.

### 3.5 파일 네이밍 규칙

- 기획서 확정 후 기술 스택에 맞춰 구체적으로 결정한다.
- 일관된 네이밍 패턴을 전체 프로젝트에 적용한다.

## 4. 리뷰 기준

에이전트 산출물 검증 및 코드 리뷰 시 아래 기준을 적용한다.

### 4.1 필수 통과 기준 (불합격 시 반려)

| # | 항목 | 설명 |
|---|---|---|
| R1 | **기능 정합성** | 기획서/작업 지시에 명시된 요구사항을 빠짐없이 구현했는가? |
| R2 | **빌드/실행 가능성** | 에러 없이 빌드되고 실행되는가? |
| R3 | **보안** | XSS, 인젝션, 하드코딩된 시크릿 등 보안 취약점이 없는가? |
| R4 | **기존 코드 비파괴** | 기존 기능이 깨지지 않았는가? |
| R5 | **커밋 컨벤션** | 커밋 메시지가 규칙을 따르는가? |
| R6 | **가독성** | 코드가 명확하고 읽기 쉬운가? 의미 불명확한 변수/함수명이 없는가? |
| R7 | **중복 제거** | 불필요한 반복 코드가 있는가? 공통 로직이 분리되어 있는가? |
| R8 | **성능** | 명백한 성능 저하 요소(불필요한 반복, 메모리 누수 등)가 없는가? |
| R9 | **일관성** | 프로젝트의 기존 패턴/스타일/네이밍과 일치하는가? |

### 4.2 리뷰 프로세스

1. 에이전트 작업 완료 즉시 생성/변경된 모든 파일을 읽는다.
2. 필수 기준(R1~R9)을 순서대로 점검한다.
3. 하나라도 불합격이면 구체적 사유와 함께 수정 지시를 내린다.
4. 모든 기준을 통과할 때까지 반복한다.
5. 검증 결과를 사용자에게 보고한다.

## 5. 에이전트 할당 계획

> 기획서 확정 후 작성 예정

에이전트 할당 시 고려 사항:
- 각 에이전트는 하나의 독립된 작업 단위를 담당한다.
- 의존성이 없는 작업은 병렬로 할당한다.
- 의존성이 있는 작업은 선행 작업 완료 후 순차 할당한다.
- 각 에이전트에게 명확한 입력(컨텍스트)과 기대 출력(산출물)을 전달한다.

### 5.1 문서 관리 및 에이전트 프롬프트 원칙

- 각 주요 디렉토리(모듈/기능 단위)에 해당 영역의 설계와 규칙을 담은 문서를 배치한다.
- 에이전트 할당 시, 프롬프트에 **해당 작업에 필요한 문서 경로만 명시적으로 포함**한다.
- 에이전트가 불필요한 문서를 읽지 않도록, 전체 문서 목록이 아닌 관련 문서만 지정한다.
- 문서 경로 예시: `docs/timer/DESIGN.md`, `docs/api/SPEC.md` 등 (기획서 확정 후 구체화)

**프롬프트 작성 템플릿:**
```
[작업 지시]
- 작업 내용: ...
- 참고 문서: <해당 작업에 필요한 문서 경로 목록>
- 기대 산출물: ...
- 제약 조건: ...
```
