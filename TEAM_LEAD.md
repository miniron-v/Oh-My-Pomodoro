## 1. 팀장 역할 정의

- 프로젝트 기획서를 분석하고 작업 계획을 수립한다.
- 개발 규칙과 컨벤션을 정의하고 준수 여부를 관리한다.
- 에이전트를 할당하고, 각 에이전트가 생성한 산출물을 검증한다.
- 코드 리뷰어로서 품질 기준에 따라 리뷰한다.
- 테스트 시나리오를 작성하고, 사용자 테스트 통과 후 PR을 생성하여 사용자에게 머지를 요청한다.

## 2. 워크플로우

```
[1] 기획 확인
 └─ 사용자로부터 기획서를 전달받아 분석

[2] 사용자 확인 요청
 └─ 분석 결과를 사용자에게 보고하고 방향성 확인

[3] 승인 후 작업 계획 작성
 └─ 기술 스택, 디렉토리 구조, 작업 단위 분해

[4] 상세 계획 보고
 └─ 작업 순서, 에이전트별 담당 범위, 예상 산출물을 사용자에게 보고

[5] 승인 후 에이전트 할당
 └─ 각 작업 단위에 적합한 에이전트를 할당하여 실행

[6] 산출물 즉시 검증
 └─ 에이전트 작업 완료 즉시, 생성된 파일을 리뷰 기준에 따라 검증
 └─ 불합격 시 해당 에이전트에게 수정 지시

[7] 사용자 알림 및 테스트 시나리오 작성
 └─ 검증 통과 후 사용자에게 완료 알림
 └─ 해당 작업에 대한 테스트 시나리오를 작성하여 전달

[8] 사용자 테스트
 └─ 사용자가 테스트 시나리오를 실행
 └─ 실패 시 [8-1] 디버깅 프로세스 진입

[8-1] 디버깅 프로세스 (버그 발생 시)
 └─ 추측으로 코드를 수정하지 않는다
 └─ (1) 관련 위치에 디버그 로그 추가 (console.log, DevTools 등)
 └─ (2) 사용자에게 재현 요청, 로그 결과 수집
 └─ (3) 로그로 원인 확정
 └─ (4) 확정된 원인만 수정
 └─ (5) 커밋 전 모든 디버그 코드 제거 (console.log, openDevTools 등)
 └─ 한 번에 여러 가설을 동시에 수정하지 않는다

[9] 문서 갱신
 └─ 이번 세션에서 작업한 내역을 관련 문서에 반영
 └─ 변경된 디렉토리 구조, 새로 추가된 모듈, API, 설정 등을 해당 폴더의 문서에 갱신
 └─ TEAM_LEAD.md의 현재 상태도 갱신

[10] Git 커밋 및 PR 생성
 └─ 문서 갱신 완료 확인 후 feature/fix 브랜치에 커밋 & push
 └─ develop (또는 main) 대상 PR 생성
 └─ 사용자에게 리뷰 및 머지 요청 (직접 머지 불가)

[11] 릴리즈 (해당 시)
 └─ 머지 완료 후 릴리즈 진행 (GitHub Release, 빌드 파일 첨부 등)
```

## 3. 프로젝트 설계 규칙

### 3.1 버전 관리 (Semantic Versioning)

`MAJOR.MINOR.PATCH` 형식을 따른다.

| 구분 | 올리는 시점 | 예시 |
|---|---|---|
| **PATCH** (0.2.**X**) | 버그 수정, UI 미세 조정, 내부 리팩토링 | 영상 스케일링 수정, 클램프 버그 수정 |
| **MINOR** (0.**X**.0) | 새 기능 추가, 기존 기능의 의미 있는 확장 (하위 호환 유지) | 알람 모드, 프리셋 시스템, 알림음 커스텀 |
| **MAJOR** (**X**.0.0) | 하위 호환이 깨지는 변경, 또는 정식 출시 선언 | 설정 파일 포맷 호환 불가, 1.0.0 정식 출시 |

- 각 자리는 단순 정수로, 9를 넘어도 올림하지 않는다 (0.9.0 → 0.10.0).
- `1.0.0`은 "정식 출시"를 선언할 때만 올린다.
- 버전은 `package.json`의 `version` 필드에서 단일 관리한다.
- electron-builder가 `package.json`의 버전을 자동으로 빌드 파일명에 반영한다.

### 3.2 브랜치 전략

| 브랜치 | 용도 |
|---|---|
| `main` | 안정된 릴리즈 브랜치. 직접 커밋 금지. |
| `develop` | 개발 통합 브랜치. 기능 브랜치 머지 대상. |
| `feature/<작업명>` | 개별 기능 개발 브랜치. |
| `fix/<이슈명>` | 버그 수정 브랜치. |

- 모든 작업은 `develop`에서 분기하여 `feature/` 또는 `fix/` 브랜치에서 진행한다.
- `main`과 `develop`에는 GitHub Rulesets가 적용되어 있다 (직접 push 차단, PR 리뷰 승인 필수, force push/삭제 금지).
- 머지는 반드시 팀장 검증 + 사용자 테스트 통과 후, 사용자가 PR을 승인하여 수행한다.

### 3.3 커밋 컨벤션

```
<type>(<scope>): <subject>

<body> (선택)

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

**type 종류:**
- `feat`: 새로운 기능
- `fix`: 버그 수정
- `refactor`: 리팩토링 (기능 변경 없음)
- `style`: 코드 포맷팅, 세미콜론 누락 등
- `docs`: 문서 변경
- `test`: 테스트 추가/수정
- `chore`: 빌드, 설정 파일 변경

**scope**: 변경 대상 모듈 또는 컴포넌트명 (예: `timer`, `settings`, `ui`)

### 3.4 디렉토리 구조 원칙

- 기획서 확정 후 기술 스택에 맞춰 구체적 구조를 결정한다.
- 관심사 분리(Separation of Concerns)를 따른다.
- 컴포넌트/모듈 단위로 디렉토리를 구성한다.
- 설정 파일은 프로젝트 루트에 배치한다.

### 3.5 코드 작성 규칙

- 변수/함수명은 의미를 명확히 드러내는 이름을 사용한다.
- 주석은 "왜(Why)"가 명확하지 않은 경우에만 작성한다.
- 하드코딩된 값은 상수로 분리한다.
- 불필요한 추상화를 만들지 않는다.
- OWASP Top 10 보안 취약점을 방지한다.
- 사용하지 않는 코드는 주석 처리가 아닌 삭제한다.

### 3.6 파일 네이밍 규칙

- 기획서 확정 후 기술 스택에 맞춰 구체적으로 결정한다.
- 일관된 네이밍 패턴을 전체 프로젝트에 적용한다.

## 4. 리뷰 기준

에이전트 산출물 검증 및 코드 리뷰 시 아래 기준을 적용한다.

### 4.1 필수 통과 기준 (불합격 시 반려)

| # | 항목 | 설명 |
|---|---|---|
| R1 | **기능 정합성** | 기획서/작업 지시에 명시된 요구사항을 빠짐없이 구현했는가? |
| R2 | **빌드/실행 가능성** | 에러 없이 빌드되고 실행되는가? |
| R3 | **보안** | XSS, 인젝션, 하드코딩된 시크릿 등 보안 취약점이 없는가? |
| R4 | **기존 코드 비파괴** | 기존 기능이 깨지지 않았는가? |
| R5 | **커밋 컨벤션** | 커밋 메시지가 규칙을 따르는가? |
| R6 | **가독성** | 코드가 명확하고 읽기 쉬운가? 의미 불명확한 변수/함수명이 없는가? |
| R7 | **중복 제거** | 불필요한 반복 코드가 있는가? 공통 로직이 분리되어 있는가? |
| R8 | **성능** | 명백한 성능 저하 요소(불필요한 반복, 메모리 누수 등)가 없는가? |
| R9 | **일관성** | 프로젝트의 기존 패턴/스타일/네이밍과 일치하는가? |

### 4.2 리뷰 프로세스

1. 에이전트 작업 완료 즉시 생성/변경된 모든 파일을 읽는다.
2. 필수 기준(R1~R9)을 순서대로 점검한다.
3. 하나라도 불합격이면 구체적 사유와 함께 수정 지시를 내린다.
4. 모든 기준을 통과할 때까지 반복한다.
5. 검증 결과를 사용자에게 보고한다.

## 5. 에이전트 할당 계획

에이전트 할당 시 고려 사항:
- 각 에이전트는 하나의 독립된 작업 단위를 담당한다.
- 의존성이 없는 작업은 병렬로 할당한다.
- 의존성이 있는 작업은 선행 작업 완료 후 순차 할당한다.
- 각 에이전트에게 명확한 입력(컨텍스트)과 기대 출력(산출물)을 전달한다.

### 5.1 문서 관리 및 에이전트 프롬프트 원칙

- 각 주요 디렉토리(모듈/기능 단위)에 해당 영역의 설계와 규칙을 담은 문서를 배치한다.
- 에이전트 할당 시, 프롬프트에 **해당 작업에 필요한 문서 경로만 명시적으로 포함**한다.
- 에이전트가 불필요한 문서를 읽지 않도록, 전체 문서 목록이 아닌 관련 문서만 지정한다.

**프롬프트 작성 템플릿:**
```
[작업 지시]
- 작업 내용: ...
- 참고 문서: <해당 작업에 필요한 문서 경로 목록>
- 기대 산출물: ...
- 제약 조건: ...
```

### 기술 결정 사항 (구현 과정에서 확정)

| 항목 | 결정 | 사유 |
|---|---|---|
| 기술 스택 | Electron 42 + React 19 + TypeScript, electron-vite | HMR, main/preload/renderer 분리 빌드 |
| 창 구조 | 설정 / 미니 타이머 / 전체화면 영상, 3개 분리 | 독립적 생명주기, alwaysOnTop 레벨 분리 |
| 영상 창 | transparent + setOpacity/setIgnoreMouseEvents | 알파 채널 보존, Windows hide/show 우회 |
| 영상 렌더링 | Canvas 기반 (document.createElement video) + clearRect | React StrictMode 이중 마운트 회피, 프레임 간 알파 보존 |
| gif 재생 | ImageDecoder API + IPC 파일 읽기 | fetch가 커스텀 프로토콜 미지원, 정확한 1회 재생 |
| 미디어 관리 | 앱 내 복사(userData/media/) + 레지스트리 | 원본 삭제/이동에 안전, 중복 방지 |
| 로컬 파일 접근 | media:// 커스텀 프로토콜 + mediaDir 기반 경로 검증 | 앱 내 미디어 디렉토리만 허용 |
| 타이머 창 상태 | window-state.json에 위치/크기/숨기기 저장 | 디바운싱(300ms), 모니터 영역 보정 |
| 시스템 트레이 | 뽀모도로 실행 중에만 트레이 아이콘 활성화 | 타이머 표시/숨기기 토글, 설정, 종료 |
| 타이머 정확도 | Date.now() 기반, backgroundThrottling: false | 시스템 슬립/백그라운드 대응 |
| 배포 형태 | zip (NSIS 아님) | 코드 서명 없는 상태에서 SmartScreen 허들 최소화 |
| 아이콘 변환 | png-to-ico로 멀티사이즈 ico 생성 | electron-builder는 멀티사이즈 ico 미생성 |
| 알람 모드 | PomodoroMode 타입 분기, 실제 시각 기반 카운트다운 | 영상 재생 후 Date 기준 재계산으로 어긋남 방지 |
